# Style Iconoclast — Developer Handover Guide

This document explains everything a developer needs to know to take over this project, run it locally, and operate the (now self-hosted) infrastructure.

The Manus-platform integration was replaced with standard equivalents on **2026-05-08**:

| Concern | Was (Manus) | Now |
|---|---|---|
| Database | TiDB (MySQL) | **Supabase Postgres** (Drizzle `pg-core`) |
| Auth | Manus OAuth + signed JWT cookie | **Supabase Auth** (Google OAuth, PKCE, browser-managed session) |
| File storage | Manus S3 + Forge presigned URLs | **Cloudflare R2** via `@aws-sdk/client-s3` |
| LLM | Manus Forge (`gemini-2.5-flash`) | **OpenAI SDK** (default `gpt-4o-mini`) |
| Notifications | Manus push (`webdevtoken.v1.WebDevService`) | **Resend** (email) |

The remaining Manus-only modules (`server/_core/imageGeneration.ts`, `_core/map.ts`, `_core/voiceTranscription.ts`, `_core/dataApi.ts`) still reference the Manus Forge endpoints. They keep working only if you set `BUILT_IN_FORGE_API_URL` / `BUILT_IN_FORGE_API_KEY`. Otherwise the corresponding features (image generation, Google Maps proxy, voice transcription, data APIs) will throw at call time.

## Project Overview

**Style Iconoclast** is a digital wardrobe management application built with:

| Layer | Technology |
|---|---|
| Frontend | React 19 + Vite + Tailwind CSS 4 + shadcn/ui |
| API layer | tRPC 11 (end-to-end type-safe RPC) |
| Backend | Express 4 (Node.js) |
| Database ORM | Drizzle ORM (`pg-core` dialect) |
| Database | Postgres (Supabase) |
| Auth | Supabase Auth (Google OAuth, PKCE) |
| File storage | Cloudflare R2 (S3-compatible) |
| Email | Resend |
| LLM | OpenAI |
| Maps | Google Maps via Manus proxy (legacy — see notes) |

---

## Key Features

- **Wardrobe** — add, tag, and organise clothing items with photos
- **Outfits** — build multi-piece looks, track cost-per-wear, share via public link
- **Build Your Look** — AI-assisted outfit builder
- **Travel Planner** — plan outfits day-by-day for trips, share trip itineraries
- **Designers** — browse and follow designer pages
- **Style Index** — analytics: cost-per-wear, category breakdown, most-worn pieces
- **Wishlist / Cart** — save items to buy later
- **Sharing** — public links for outfits and trips with lightbox thumbnails

## Running Locally

```bash
# 1. Install dependencies
pnpm install

# 2. Set up environment variables
cp .env.example .env
# Fill DATABASE_URL, VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY,
# SUPABASE_SERVICE_ROLE_KEY, R2_*, OPENAI_API_KEY, RESEND_API_KEY, etc.

# 3. Push the Postgres schema to Supabase
pnpm db:push   # runs drizzle-kit generate + drizzle-kit migrate

# 4. Start dev server (frontend + backend on port 3000)
pnpm dev
```

### Supabase setup checklist

1. Create a project at https://supabase.com/dashboard
2. **Project Settings → Database** — copy the *Transaction* pooler URL (port 6543) into `DATABASE_URL`
3. **Project Settings → API** — copy `URL` and `anon` and `service_role` keys
4. **Authentication → Providers → Google** — enable, paste your Google OAuth client ID/secret (create at https://console.cloud.google.com → APIs & Services → Credentials → OAuth 2.0 Client)
5. **Authentication → URL Configuration** — add redirect URLs: `http://localhost:3000/auth/callback` and your production URL

### Cloudflare R2 setup checklist

1. Cloudflare Dashboard → R2 → Create bucket (name = `R2_BUCKET`, default `style-iconoclast`)
2. R2 → Manage R2 API Tokens → Create token with read+write scope on the bucket — yields `R2_ACCESS_KEY_ID` and `R2_SECRET_ACCESS_KEY`
3. Find your account ID in the Cloudflare sidebar — this is `R2_ACCOUNT_ID`
4. (Optional) Bind a custom domain to the bucket and set `R2_PUBLIC_URL` to skip signed URL generation

### OpenAI / Resend

- OpenAI: https://platform.openai.com/api-keys
- Resend: https://resend.com → verify your sending domain (DNS records required), then create an API key. Set `NOTIFY_FROM_EMAIL` to a verified-domain address.

## Database Schema

Tables (in `drizzle/schema.ts`):

| Table | Purpose |
|---|---|
| `users` | User accounts. `openId` is the Supabase user UUID. |
| `wardrobe_items` | Individual clothing/accessory items |
| `item_tags` | User-defined tags on items |
| `price_history` | Price snapshots over time |
| `outfits` | Named outfits with shareToken |
| `outfit_items` | Many-to-many: outfits ↔ items, with slot |
| `cart_items` | Shopping cart |
| `designers_shops` | Designer / shop directory |
| `trips` | Travel trips with dates, destination, shareToken |
| `trip_days` | Per-day outfit assignments (2 outfits/day, weather, notes) |
| `packing_items` | Per-trip packing checklist |
| `collaborators` | Wardrobe sharing invites |

To regenerate migrations after a schema change:
```bash
pnpm db:push
```

## Auth Flow

1. User clicks "Continue with Google" on `/login` → `signInWithGoogle()` calls `supabase.auth.signInWithOAuth({ provider: 'google' })`
2. Browser is redirected to Google → back to `/auth/callback?code=...`
3. `AuthCallback.tsx` calls `supabase.auth.exchangeCodeForSession(code)` — Supabase JS stores the session in localStorage (PKCE)
4. `useAuth.ts` subscribes to `supabase.auth.onAuthStateChange` and invalidates `trpc.auth.me`
5. Every tRPC request includes `Authorization: Bearer <access_token>` (set in `main.tsx`)
6. Server `_core/context.ts` extracts the token, calls `supabase.auth.getUser(token)`, and upserts the user into our `users` table on first sign-in

Logout: `supabase.auth.signOut()` (clears local session) + `trpc.auth.logout` (no-op).

## Deployment

Standard Node.js Express server serving both the API (`/api/trpc`) and the built React frontend.

- **Railway / Render / Fly.io** — `pnpm build` produces `dist/`, run `pnpm start`
- **Docker** — straightforward; ensure `DATABASE_URL` and Supabase keys are set as runtime env

Don't forget to add your production URL to Supabase's Authentication → URL Configuration → Redirect URLs.
